ICO file format reference (so the executor doesn't have to look it up):
- ICONDIR header: 6 bytes — `00 00` (reserved), `01 00` (type=1, ICO), `NN 00` (image count, little-endian uint16). For two images: `00 00 01 00 02 00`.
- ICONDIRENTRY: 16 bytes per image, immediately following the header. Fields:
  - byte 0: width (1 byte; 0 means 256)
  - byte 1: height (1 byte; 0 means 256)
  - byte 2: color palette count (0 for PNG-compressed)
  - byte 3: reserved (0)
  - bytes 4-5: color planes (uint16 LE, set to 1)
  - bytes 6-7: bits per pixel (uint16 LE, set to 32)
  - bytes 8-11: image data size in bytes (uint32 LE)
  - bytes 12-15: offset of image data from start of file (uint32 LE)
- Image data: PNG-compressed bitmap (full PNG bytes including signature) is the modern, well-supported approach for entries >= 32x32 and works fine for 16x16 too. Browsers including Chromium handle PNG-in-ICO correctly.
- File layout: [6-byte ICONDIR][16-byte entry for img1][16-byte entry for img2][PNG bytes for img1][PNG bytes for img2]
- For two images, first PNG offset = 6 + 16*2 = 38.

Create `scripts/generate-favicon-ico.mjs` (Node ESM, no new dependencies — uses `sharp` already in node_modules, plus built-in `node:fs` and `node:path`).

The script must:
1. Resolve paths relative to the repo root (use `import.meta.url` + `fileURLToPath` to be cwd-independent). Source: `public/favicon.svg`. Output: `public/favicon.ico`.
2. Read the SVG bytes with `fs.readFileSync`.
3. Use `sharp(svgBuffer, { density: 384 })` (high density so the 64-viewBox SVG rasterizes crisply at small sizes — 384 DPI gives sharp's rasterizer enough resolution before downscale) to produce two PNG buffers:
   - 32x32 PNG: `sharp(svg, { density: 384 }).resize(32, 32).png({ compressionLevel: 9 }).toBuffer()`
   - 16x16 PNG: `sharp(svg, { density: 384 }).resize(16, 16).png({ compressionLevel: 9 }).toBuffer()`
4. Pack both PNG buffers into a multi-resolution ICO using a small hand-rolled packer (NO new dependency — do not add `to-ico`). The packer must:
   - Allocate a 6-byte ICONDIR header: `Buffer.from([0,0,1,0,2,0])` (reserved=0, type=ICO, count=2).
   - Build two 16-byte ICONDIRENTRY records. For each entry, write:
     - width byte (16 or 32 — note: the field is one byte and 0 means 256, but 16 and 32 fit fine)
     - height byte (same as width)
     - 0 (no palette, since PNG-compressed)
     - 0 (reserved)
     - uint16 LE color planes = 1
     - uint16 LE bits per pixel = 32
     - uint32 LE size of PNG bytes
     - uint32 LE offset to PNG bytes (computed as 6 + 16*2 + cumulative size of preceding PNG buffers)
   - Order entries small-to-large (16x16 first, then 32x32) — this is the conventional order and is what Windows/Chromium expect.
   - Concatenate: [header, entry16, entry32, png16Bytes, png32Bytes].
   - Use `Buffer.concat([...])` for the final write.
5. `fs.writeFileSync(outPath, icoBuffer)`.
6. Print a single-line confirmation: `console.log(\`Wrote \${outPath} (\${icoBuffer.length} bytes; 16x16 + 32x32)\`)`.

Implementation notes:
- The script is ESM (`.mjs`). Import: `import sharp from "sharp"`, `import fs from "node:fs"`, `import path from "node:path"`, `import { fileURLToPath } from "node:url"`.
- Wrap the body in `async function main()` and call it; surface errors via `.catch(err => { console.error(err); process.exit(1); })`.
