# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Navigation & Design System.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 2 delivers the persistent UI shell that every page shares: a sticky header with inline desktop nav and mobile hamburger, a full-screen overlay navigation for mobile, a dark/light mode toggle with FOUC prevention, a two-column footer, and consistent application of the design token system across all components. No new content pages are introduced. All token values are inherited from Phase 1's `global.css` — this phase implements them into interactive components, does not redefine them.

## Typography

Two weights in use across Phase 2 (400 Regular + 600 Semibold). No new weights introduced per CONTEXT.md D-18.

| Role | Size | Weight | Line Height | Phase 2 Usage |
|------|------|--------|-------------|---------------|

- Google Fonts URL in BaseLayout loads `wght@400;600` — no change required for Phase 2.

**Accent reserved for (complete list — Phase 2 additions in bold):**
1. **Active nav link indicator** — 2px underline beneath active page link (new in Phase 2)
2. **Nav link hover underline** — same 2px underline appears on hover for inactive links (new in Phase 2)
3. Primary CTA button fill (Phase 3+)
4. **Footer "Get in touch" link text** (new in Phase 2)
5. Keyboard focus rings (`:focus-visible` outline, 2px solid accent color, 2px offset)
6. Case study accent color overrides (Phase 3)

**Dark mode implementation:**
- FOUC prevention script (inline, synchronous, in `<head>` before CSS): reads `localStorage['theme']`, falls back to `window.matchMedia('(prefers-color-scheme: dark)')`, adds `.dark` to `<html>` if dark. Must run before any body renders (D-02).
- Toggle writes `localStorage['theme'] = 'dark'` or `'light'` and flips `.dark` class on `<html>` (D-01).
- The `.dark` class override in `global.css` is already declared (from Phase 1 — `:root.dark {}`).

## Accessibility Baseline

| Requirement | Implementation |
|-------------|---------------|
| Touch targets | All interactive elements: `min-h-[44px]` and `min-w-[44px]` |
| Landmark regions | `<header>`, `<main>`, `<footer>` — already in BaseLayout |
| Nav landmark | `<nav aria-label="Main navigation">` for header nav |
| Nav landmark (footer) | `<nav aria-label="Footer navigation">` for footer nav links |
| Overlay role | `role="dialog" aria-modal="true" aria-label="Navigation"` |
| Focus trap | Overlay traps focus within its links and close button |
| Return focus | Closing overlay returns focus to the hamburger trigger |
| Esc close | `keydown` event listener on `document` while overlay is open |
| Body scroll lock | `document.body.style.overflow = 'hidden'` on open; restored on close |
| Screen reader toggle | `aria-expanded` on hamburger button reflects overlay state |
| Reduced motion | Overlay fade animation respects `prefers-reduced-motion`: `@media (prefers-reduced-motion: reduce) { transition: none }` |

## Implementation Notes for Executor

1. **Google Fonts URL:** BaseLayout loads `wght@400;600`. No change required — Phase 2 uses only weight 400 (Regular) and weight 600 (Semibold) per D-18.

2. **FOUC prevention script placement:** The inline `<script>` must be the first child of `<head>`, before the Google Fonts `<link>` tags. Any async loading or deferral will cause a visible flash.

3. **Overlay implementation approach:** Use a `<div>` with `position: fixed; inset: 0; z-index: 60` toggled via a boolean Astro/JS state. Vanilla JS is sufficient — no framework needed.

4. **Active page detection:** In Astro, use `Astro.url.pathname` to compare against each nav link href and conditionally apply the active underline class.

5. **Footer two-column grid:** `grid grid-cols-1 md:grid-cols-2 gap-xl` — left column gets the CTA content, right column gets the stacked nav links. Social links (LinkedIn icon) should live in the left column beneath the "Get in touch" link (D-15 implementation detail deferred to planning).

6. **No new CSS files:** All component styles use Tailwind utility classes and the CSS custom properties already declared in `global.css`. Do not create component-scoped CSS files.
