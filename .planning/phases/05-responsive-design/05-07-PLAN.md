  <how-to-verify>
    Step 0 — Get the preview onto your iPhone.

    The executor will choose ONE of:
    - **Route A (preferred): Cloudflare Pages branch deploy.** The executor pushes the current branch; Cloudflare auto-builds. You open the preview URL on your iPhone Safari. Wait for the executor to share the URL (~2 min build).
    - **Route B: Local network preview.** The executor runs `npm run build && npm run preview --host` and shares a `http://<your-LAN-IP>:4321` URL. Your iPhone must be on the same Wi-Fi network as the dev machine.
    - **Route C: ngrok tunnel.** Only if Route A and B are blocked. The executor runs a tunnel and shares the tunnel URL.

    Step 1 — Mobile overlay nav (NAV-05).

    On iPhone Safari, on the home page (`/`):
    1. Tap the hamburger icon in the header. The overlay should open with a fade-in animation.
    2. **Critical test:** While the overlay is open, try to scroll the page by swiping anywhere on the screen that is not a nav link. The page background MUST NOT scroll. If it scrolls (even a tiny rubber-band bounce), this is a FAIL — record it.
    3. Tap a nav link (any one). The overlay should close and the page should navigate.
    4. Use the browser back button to return to the home page. Re-open the overlay.
    5. Tap the X close button (top-right). The overlay should close.
    6. **Critical test:** After close, the page should be exactly where you were before opening the overlay. No jump to top.
    7. Tap each of the four nav links inside the overlay one at a time (re-opening the overlay between taps). Confirm each tap registers on first try with no missed-tap retries.

    Step 2 — Case study readability (CASE-06).

    Navigate to `/projects/project-alpha` (or whichever case study slug appears in your URL bar).
    1. Read from the top of the page to the prev/next navigation at the bottom in one continuous scroll.
    2. Note any of the following — each is a new FAIL:
       - Text that wraps awkwardly (orphan words, mid-word breaks)
       - An image that crops the wrong part of the picture
       - A prev/next button that requires zooming to tap accurately
       - Any content that overflows horizontally and produces a horizontal scrollbar
       - Any image that loads visibly large (looks pixelated or oversized for the screen)

    Step 3 — Image asset sizing (RESP-02).

    This step needs a Mac with Safari Develop menu enabled (`Safari → Settings → Advanced → Show Develop menu`). Your iPhone also needs Web Inspector enabled (`Settings → Safari → Advanced → Web Inspector → on`). Connect the iPhone to the Mac via USB cable or pair via Wi-Fi.

    1. On the iPhone, open `/about` (full reload — pull-to-refresh).
    2. On the Mac: `Safari → Develop → <Your iPhone name> → about`. The Web Inspector opens.
    3. In Web Inspector, go to the Network tab. Filter by Images.
    4. Reload the iPhone page. The Network tab populates with image requests.
    5. Find the About hero photo request. Inspect its URL. The filename should contain something like `_400`, `_600`, `_800`, or `_1120` — the served width. **At 375 px DPR-2-or-3, the expected variant is 600 px or 800 px, NOT 1120 px.** If 1120 px is served, that is a FAIL.
    6. Repeat on `/projects/project-alpha`. Find the cover image request. **Expected variant is 800 px, NOT 1800 px.**

    Step 4 — Sign off (or report FAILs).

    Compile a single message with:
    - Did Step 1 pass? (yes / no — if no, describe what failed)
    - Did Step 2 pass? (yes / no — if no, list issues)
    - Did Step 3 pass? (yes / no — if no, report the served variant widths)

    If all three pass, signal `approved` and the executor will upgrade PROVISIONAL-PASS → PASS in the audit matrix and stamp the Sign-Off section.
  </how-to-verify>
