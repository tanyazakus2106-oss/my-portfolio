<objective>
Replace the current scroll-lock implementation in `MobileNav.astro` (which sets `document.body.style.overflow = 'hidden'`) with the position-fixed + scroll-position-capture pattern (RESEARCH section 4) so the overlay nav reliably prevents background scroll on iOS Safari.

Per RESEARCH section 4: `overflow: hidden` on `<body>` does NOT prevent momentum scroll on iOS Safari, and `overscroll-behavior: none` is not supported in Safari as of 2024. The robust fix is to set `position: fixed`, `top: -<scrollY>px`, `width: 100%` on `<body>` while the overlay is open, and restore the scroll position on close.

This is an internal mechanism change inside the existing `open()` / `close()` script functions. Every Phase 2 locked decision (D-09 close gestures, D-10 backdrop, D-11 focus trap, D-12 animations) is preserved unchanged. Only the two lines that toggle `document.body.style.overflow` are replaced.

Implements: NAV-05 (mobile overlay usability on real device), RESP-03 (mobile-nav touch usability — scroll bleed makes targets unreliable on iOS).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client JS to DOM style API | The script sets inline styles (`position`, `top`, `width`) on `document.body`. Only literal string and computed numeric values cross — no user-controlled input flows into these writes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | T (Tampering) | `document.body.style.top` written from `window.scrollY` | accept | `window.scrollY` is a browser-provided numeric value, not user-injected content. The template literal `` `-${scrollY}px` `` is interpolated by the JS engine, not concatenated as HTML. No XSS surface. |
| T-05-03-02 | D (Denial of service) | `position: fixed` on `<body>` while overlay is open | mitigate | If the overlay close handler ever throws (e.g., DOM exception during `parseInt`), the body would remain stuck in fixed-position. Mitigation: `parseInt(... || '0') * -1` defends against a missing or non-numeric `top` value; the close path runs the unset assignments unconditionally before `scrollTo`. Acceptable risk given the existing close paths (Esc, backdrop, link click) all invoke the same `close()` function. |
| T-05-03-03 | I (Information disclosure) | `window.scrollTo(0, scrollY)` restore | accept | Scroll position is non-sensitive UI state; reading and restoring it discloses nothing private. |

Severity: all low. No high-severity threats. `security_enforcement` gate passes.
</threat_model>
