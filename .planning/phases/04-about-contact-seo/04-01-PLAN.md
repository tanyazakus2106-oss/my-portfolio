The Header `.tz-logo` (Header.astro:23) is a CSS-styled circular badge containing the literal text "tz". The favicon must reproduce this visual identity in a viewBox tight enough to render legibly at 16x16 / 32x32 browser tab scale. Use inline `<text>` (no external font dependency — favicons cannot fetch webfonts at render time) and adopt the same prefers-color-scheme dark/light pattern the stock Astro favicon used so the favicon adapts to the user's OS theme.

**Overwrite `public/favicon.svg` with this exact content:**

```svg
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 64 64">
  <style>
    .bg { fill: #1a1a1a; }
    .fg { fill: #ffffff; }
    @media (prefers-color-scheme: dark) {
      .bg { fill: #ffffff; }
      .fg { fill: #1a1a1a; }
    }
  </style>
  <rect class="bg" width="64" height="64" rx="14"/>
  <text x="32" y="44" class="fg" font-family="-apple-system, BlinkMacSystemFont, 'Segoe UI', Helvetica, Arial, sans-serif" font-size="36" font-weight="500" text-anchor="middle" letter-spacing="-1">tz</text>
</svg>
```

Notes on the mark:
- Rounded square (`rx=14` on a 64x64 viewBox) — slightly softer than a perfect circle, reads better at 16x16 favicon scale than a circle which becomes a fuzzy disc
- Letters "tz" centered in the badge, system-sans fallback stack so the mark renders without webfonts
- `letter-spacing="-1"` mirrors the Header `letter-spacing: -0.03em`
- Dark/light fill swap via `prefers-color-scheme: dark` — matches the stock Astro pattern so the favicon adapts to the OS theme (white badge on dark UI, dark badge on light UI)
- Hex literals `#1a1a1a` and `#ffffff` are intentional here — favicon SVGs cannot reference CSS custom properties from `src/styles/global.css` (favicons load standalone, not within the page DOM). Token-only styling rule (Pattern 2) applies to component markup, not to standalone static SVG assets in `public/`.

**Do NOT touch `public/favicon.ico`.** The ICO format requires a binary encoder. Per the orchestrator's guidance: "SVG-only is acceptable for modern browsers if .ico generation is too heavy (older browsers fall back gracefully)." Modern browsers (Chrome, Firefox, Safari, Edge — all 2018+) prefer `image/svg+xml` when both link tags are present and the SVG one comes first in the `<head>`. Older browsers fall back to the existing `public/favicon.ico` (the stock Astro one) — visually inconsistent but functional. Document this trade-off in the SUMMARY: "favicon.ico left as stock Astro default; SVG covers all modern browsers; ICO replacement deferred to v2 polish or to Tanya providing a real .ico file."

When Tanya later provides a custom favicon set (proper .ico, multi-size .png, apple-touch-icon, etc.), she overwrites `public/favicon.svg` and `public/favicon.ico` — no markup change needed because BaseLayout link tags reference fixed paths.
