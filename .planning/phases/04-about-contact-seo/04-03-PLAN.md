**Primary method (base64-decoded, known-valid minimal PDF — use this):**

A pre-validated minimal-PDF byte string is decoded directly. This avoids the byte-offset pitfalls of hand-writing the `xref` table (the previous version of this plan had approximate offsets which can fail validation in strict PDF viewers). The base64 below decodes to a structurally valid 1-page PDF: `%PDF-1.4` header, catalog object, pages object, single empty page object, correctly-aligned `xref` table, `trailer`, `startxref`, and `%%EOF` terminator.

This produces a ~190-byte structurally valid PDF that opens cleanly in Preview, Acrobat, Chrome's PDF viewer, and Firefox's PDF viewer. It renders as a single blank Letter-size page — clearly a placeholder, but unambiguously a valid PDF. Tanya replaces the file with her real resume before launch.

The descriptive filename (`tanya-zakus-designer-resume.pdf`) means saved/forwarded copies carry her name (per the `<specifics>` "Resume filename SEO benefit" note).

**Historical reference (do NOT use):** A prior version of this plan attempted to hand-write the PDF in Node with manually computed `xref` byte offsets. This is fragile — the offsets must match the actual byte position of each `obj` declaration, and any whitespace difference breaks validation. The base64 method above is preferred because the offsets are pre-validated; the executor does not need to compute them.

**Cross-platform note on `base64 -d`:**
- Linux (`coreutils`): `base64 -d` works.
- macOS (BSD): `base64 -d` works on modern macOS (12+); `base64 -D` is the older flag if `-d` fails.
- The executor should try `base64 -d` first; fall back to `base64 -D` if `-d` returns "invalid option".
